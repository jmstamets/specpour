Re-walk findings, priority order:
F1 — MFA enrollment broken in real browser (blocker; section (c) is untestable past step 2). "Set up two-factor authentication" fails with the raw ASP.NET minimal-API error "Implicit body inferred for parameter 'request' but no body was provided" — the web client is POSTing with no body to an endpoint whose signature infers a required JSON body. Diagnose why the contract/acceptance suites pass while the browser fails (almost certainly: the C# test client sends {} where the generated Dart client sends nothing for an empty-request POST) — then fix on the correct side: if the operation genuinely needs no input, the endpoint signature should take no body parameter and the OpenAPI schema should say so; don't paper over it client-side. This is green-but-dead #2.
F2 — sign-in state machine defect (high; trust-damaging). Reproduced: failed sign-in shows "Invalid email or password"; returning to the sign-in page shows stale form data; submitting fails and the button becomes permanently disabled even after correcting input — yet navigating back to home reveals the user is actually signed in. So a sign-in succeeded without the UI transitioning, the submitting/disabled state never resets on error, and form state isn't cleared. Audit the auth flow's state management end-to-end: success must navigate unconditionally, failure must re-enable submission, and stale credentials must not persist across entries. Acceptance coverage: fail-then-succeed sequence in the headless-Chrome tier.
F3 — session not persisted across page reload (blocks (f) verification; ruling included). Refreshing the browser logs the user out — tokens are evidently memory-only. Ruling: implement refresh-token rotation (OpenIddict one-time-use refresh tokens — enable if not already) with platform-appropriate persistent storage: secure storage on native; on web, persisted storage with the XSS exposure explicitly documented and a CSP hardening task filed as a T139 launch-gate rider (a strict CSP is the compensating control). On app start: silent refresh → restored session; refresh failure → clean signed-out state, never an error. The BFF/cookie-session alternative is noted and rejected for V1 as disproportionate — record both in an ADR. The 10-minute access-token decision (T166 context) makes this non-optional: short tokens without working refresh persistence would mean sign-in every 10 minutes.
F4 — data export must produce a portable artifact. "Export my data" currently renders on-screen only, unselectable. The lifecycle requirement is data portability: export must download a machine-readable file (JSON) containing the export payload; the on-screen view is a courtesy rendering and becomes selectable via T172's work. Extend the (h) acceptance test: export produces a downloadable artifact whose contents match the API payload.
Structural item — grow the browser tier deliberately. Three green-but-dead instances (registration, MFA enroll, and arguably session persistence) all share one shape: correct backend + passing suites + broken real-browser behavior. The headless-Chrome tier you built for #1 is the answer; extend it now to a deliberate smoke set covering the auth-critical journeys — register, sign in (fail-then-succeed), MFA enroll→confirm→disable, reload-restores-session, export download — and wire it into the CI task already filed. This tier is now load-bearing; treat additions to it as part of any task touching auth or the web client.

Reload resumes, never duplicates: after F3, a browser reload silently restores the same session — the sessions list shows an unchanged count and the same session identity (assert session ID stable across reload in the headless-Chrome tier), with last-active updated. A new entry on reload is a test failure.
Orphan hygiene: define and implement session-expiry semantics — a session whose refresh token is expired or consumed-without-rotation must not render as "active." Confirm the refresh-token lifetime is explicitly configured (not silently defaulted), state the chosen value, and have the sessions list either filter or visibly age-out dead sessions. Include cleanup for the orphans this bug already created in dev data.
Revocation linkage sanity: since T166's denylist work will key off sessions, confirm the restored-session identity is the same one TryRevokeAsync targets — resume must not bypass a revocation issued while the client was closed (test: revoke session from browser B while browser A is closed; A's next launch must land signed out, not silently resumed).
